## 🎯 نظرة عامة

### ✨ الهدف الرئيسي
مشروع **VisionSafe360** هو نظام مراقبة ذكي يعتمد على الذكاء الاصطناعي لكشف السقوط والأخطار الأمنية في بيئات العمل بالوقت الفعلي. يهدف إلى:
- ✅ الكشف الفوري عن السقوط (fall detection)
- ✅ تنبيه الفريق الطبي خلال 30 ثانية
- ✅ تقليل معدل الوفيات الناتجة عن السقوط
- ✅ الامتثال للمتطلبات القانونية والتنظيمية

### 📊 الإحصائيات المهمة
- **700 وفيات** سنوياً من السقوط في صناعة البناء في الولايات المتحدة
- **27%** من إصابات العمل بسبب الانزلاقات والسقوطات على نفس المستوى
- **10% انخفاض** في البقاء للحياة مع كل دقيقة تأخير في التدخل الطبي
- **30 ثانية** هي المهلة الزمنية الحرجة للاستجابة

## 🔍 ما هو كشف السقوط؟

### تعريف بسيط
كشف السقوط هو عملية **تتبع حركة الشخص في الفيديو** والتعرف على اللحظة التي ينتقل فيها من وضع طبيعي (واقف) إلى وضع السقوط (ملقى على الأرض) **بشكل مفاجئ وغير متعمد**.

### أنواع السقوط المكتشفة

| النوع | الوصف | الخطورة | الأعراض |
|-------|-------|---------|---------|
| **الانهيار المفاجئ** | سقوط فوري بسبب جلطة أو اختناق حراري | ⚠️ حرج | حركة عمودية سريعة لأسفل |
| **السقوط من التعثر** | الشخص يتعثر ويسقط على الأرض | ⚠️ مرتفع | حركة مائلة سريعة |
| **السقوط التدريجي** | انزلاق بطيء للأرض (الإرهاق) | ⚠️ مرتفع | حركة تنازلية ببطء |
| **السقوط من ارتفاع** | سقوط من سلم أو سقالة | ⚠️ حرج | عدم ظهور الشخص بعد |

### الأحداث التي يجب عدم اعتبارها سقوط ❌

| الحدث | السبب |
|--------|--------|
| الانحناء | نية متعمدة لالتقاط شيء |
| الجلوس | وضع طبيعي وآمن |
| الاستلقاء المتعمد | راحة مخطط لها |
| الزحف | حركة متحكم بها في مساحات ضيقة |
| الركوع | موقف عمل شرعي |

## ⚙️ كيف يعمل النظام؟

### 1️⃣ المرحلة الأولى: التقاط الفيديو والكشف عن الأشخاص

```
┌─────────────────────────────────────────┐
│   كاميرا المراقبة 🎥                  │
│   (فيديو مباشر أو ملف)                 │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│  1. قراءة الفيديو (StreamHandler)      │
│     • معدل العينات: 15.5 FPS          │
│     • دقة: متوافقة مع الكاميرا         │
│     • المعالجة: في الوقت الفعلي 🔄    │
└─────────────────────────────────────────┘
```

### 2️⃣ المرحلة الثانية: استخراج البيانات الحركية (Pose Estimation)

```
┌─────────────────────────────────────────┐
│   نموذج الهيكل العظمي 🦴             │
│   (YOLO11n-Pose مع fallback إلى YOLOv8) │
├─────────────────────────────────────────┤
│   المخرجات:                            │
│   • 17 نقطة رئيسية (keypoints)        │
│   • مفاصل: الرأس، الكتفين، الوركين     │
│   • الكوعين، الركبتين، الكاحلين        │
│   • معامل ثقة لكل نقطة (0-1)         │
└─────────────────────────────────────────┘
```

#### النقاط الرئيسية المستخرجة (COCO-17):
```
0: الأنف          1-2: العيون
3-4: الآذان       5-6: الكتفين
7-8: الكوعين      9-10: المعاصم
11-12: الوركين    13-14: الركبتين
15-16: الكاحلين
```

### 3️⃣ المرحلة الثالثة: تحليل الحركة والزمن

```python
# حساب المؤشرات الرئيسية:

1. نسبة العرض إلى الارتفاع (w/h)
  ├─ المنخفضة (< 0.8 غالبا) = وضع قائم/عمودي ✓
  └─ العالية (> 0.85) = اشتباه وضع أفقي/سقوط ⚠️

2. السرعة الرأسية (Vertical Velocity)
  ├─ إذا dy/dt > 15 px/frame مع شكل أفقي = سقوط محتمل 🚨
  └─ الحركة البطيئة أو بدون شروط الشكل = لا يتم تأكيد سقوط ✓

3. المساحة المحتلة (Bounding Box Area)
  ├─ تستخدم كفحص ثبات إضافي عند غياب pose
  └─ ليس شرطا وحيدا لإطلاق السقوط

4. نسبة الوركين (Hip Ratio)
  ├─ hip_ratio < 0.2 = دخول مرشح سقوط
  └─ hip_ratio > 0.6 = تعافي/خروج من حالة السقوط
```

### 4️⃣ المرحلة الرابعة: آلة الحالات (State Machine)

```
                    ┌──────────────┐
                    │   حالة طبيعية │
                    │   (Normal)   │
                    └──────┬───────┘
                           │
             سرعة نزول سريعة ▼
                    ┌──────────────────┐
                    │   حالة مرشحة      │
                    │   (Candidate)    │
                    │   (تأكيد بعد 2.0 ثانية) │
                    └──────┬───────────┘
                           │
        تأكيد: الشخص ملقى▼
        + عدم حركة
                    ┌──────────────────┐
                    │   مؤكد (Confirmed)│
                    │   🚨 تنبيه!      │
                    └──────────────────┘
```

## 🔧 التقنيات المستخدمة

### 1. نماذج الذكاء الاصطناعي 🤖

#### أ) نموذج الهيكل العظمي (Pose Model)
```yaml
المستخدم: YOLO11n-Pose (fallback: YOLOv8n-Pose)
المزايا:
  • سرعة عالية (10.7 FPS inference)
  • دقة جيدة في الكشف (100% track coverage)
  • منخفضة الموارد (151 MB VRAM)
  • يعمل على أجهزة الحد الأدنى

المخرجات:
  • 17 keypoint لكل شخص مكتشف
  • معامل ثقة لكل keypoint
```

#### ب) نموذج تتبع الشخص (Tracker)
```yaml
البرنامج: ByteTrack
الهدف: ربط الأشخاص بين الإطارات المتتالية
المزايا:
  • لا يفقد الشخص عند الانسداد الجزئي
  • معرّف ثابت (Track ID) للشخص عبر الوقت
  • سرعة معالجة عالية
```

### 2. معالجة البيانات 📊

```python
# قائمة الانتظار والحالة الزمنية
├─ centroid_history: مراكز الأشخاص
├─ aspect_ratio_history: نسب الارتفاع/العرض
├─ area_history: المساحات المحتلة
└─ velocity_history: السرعات

# STREAM_BUFFER_SIZE الافتراضي = 1 (latest-frame policy)
# نافذة حساب السرعة FALL_VELOCITY_WINDOW = 8 إطارات
# الهدف: تأخير منخفض + تحليل زمني قصير ومستقر
```

### 3. المعالجة المتوازية ⚡

```
Thread 1: StreamHandler (معالج البث)
  └─ قراءة الفيديو + عصر الإطار
  └─ استدعاء نموذج الهيكل العظمي (GPU)

Thread 2: Main Loop (الحلقة الرئيسية)
  └─ معالجة الكشف والتتبع
  └─ تحليل الحركة (HazardAnalyzer - CPU فقط)
  └─ إرسال التنبيهات
```

### 4. قاعدة البيانات 💾

```yaml
Database: رسالة في قائمة الانتظار (في الذاكرة)
التخزين:
  • كل حدث سقوط مؤكد
  • الطابع الزمني (timestamp)
  • معرّف الكاميرا
  • معرّف المسار

النقل إلى الخادم:
  • عند الاتصال بالإنترنت
  • فاشلة يتم إعادة المحاولة
```

## 📏 معايير الكشف

### المعاملات الحرجة (Critical Thresholds)

```yaml
# ✨ نسخة محسّنة (v2) - تقليل الإيجابيات الكاذبة

# معايير نسبة الحجم (Aspect Ratio)
FALL_ASPECT_RATIO_THRESHOLD: 1.0  # ENHANCED: أصعب (كان 0.85)
                                  # فقط حقاً horizontal أو قريب منها
                                  
FALL_UPRIGHT_THRESHOLD: 0.65      # NEW: تعريف الوضع "القائم"
                                  # إذا كان > 0.65 فالشخص ليس قائماً

ASPECT_RATIO_CHANGE_THRESHOLD: 0.25  # NEW: أدنى تغيير مفاجئ
                                      # يجب أن يتغير من 0.65 إلى 0.9+

# معايير الوركين (Hip-Based)
FALL_HIP_RATIO_THRESHOLD: 0.15    # TIGHTER: أصعب (كان 0.2)
                                  # الوركين يجب أن تكون قريبة جداً من القدمين

FALL_HIP_RECOVERY_THRESHOLD: 0.55 # TIGHTER: (كان 0.6)
FALL_HIP_CONFIDENCE_MIN: 0.7      # NEW: ثقة عالية جداً (كان 0.5)
                                  # بيانات الوركين يجب أن تكون موثوقة

# معايير السرعة والتسارع
FALL_VELOCITY_THRESHOLD: 20.0     # INCREASED: 15 → 20 px/frame
                                  # يجب حركة نزولية سريعة جداً

FALL_VELOCITY_WINDOW: 6           # REDUCED: 8 → 6 إطارات
                                  # كشف أسرع، أقل تأخير

FALL_ACCELERATION_THRESHOLD: 3.0  # NEW: التسارع السلبي المطلوب
                                  # السقوط = تسارع سريع (ليس سرعة ثابتة فقط)

FALL_HORIZONTAL_MOVEMENT_MAX: 10.0 # NEW: حركة أفقية قصوى
                                   # السقوط المفاجئ = حركة عمودية فقط
                                   # الجلوس = قد تكون حركة أفقية أكبر

# معايير الاستقرار
FALL_CANDIDATE_TIMEOUT: 2.5       # INCREASED: 2.0 → 2.5 ثانية
                                  # تجنب الإنذارات المتسرعة

FALL_STABILITY_FRAMES: 4          # NEW: يجب البقاء "ساقط" لـ 4 إطارات
                                  # قبل تأكيد الكشف

FALL_IMMOBILITY_THRESHOLD: 3.0    # TIGHTER: 5.0 → 3.0 بكسل
                                  # يجب أن يكون الشخص ثابتاً جداً

FALL_AREA_JITTER_THRESHOLD: 0.10  # TIGHTER: 0.15 → 0.10
                                  # تحمل اهتزاز أقل
```

  الافتراضي: ./output
  الوصف: مجلد حفظ السجلات والنتائج
```

## 🎯 كيف يعمل الفرق العملي؟

### مثال #1: جلوس مائل ❌ (كان يعطي false positive)

**الموقف**: شخص جالس يميل إلى الأمام (مثل استخدام اللابتوب)

**الحركة**:
- الإطار 1: جالس (aspect_ratio = 0.55, velocity = 0, accel = 0)
- الإطار 2: انحنى (aspect_ratio = 0.72, velocity = 4px↓, accel = 4)
- الإطار 3: مائل أكثر (aspect_ratio = 0.88, velocity = 6px↓, accel = 2)

| الإصدار | النتيجة | السبب |
|--------|--------|------|
| **v1** | 🔴 False Positive | aspect_ratio > 0.85 |
| **v2** | ✅ صحيح (لا إنذار) | • acceleration ≪ 3.0 ❌ • horizontal_movement كبير ❌ • pattern بطيء التطور |

### مثال #2: سقوط فعلي حقيقي ✅

**الموقف**: شخص وقف ثم سقط للأرض

**الحركة**:
- الإطار 1: واقف (aspect_ratio = 0.45, velocity = 0, accel = 0)
- الإطار 2: فقدان التوازن (aspect_ratio = 0.65, velocity = 25px↓, accel = 25)
- الإطار 3: في الهواء (aspect_ratio = 0.95, velocity = 35px↓, accel = 10)
- الإطار 4-6: على الأرض (aspect_ratio = 1.05+, velocity = 2px↓, accel = -15)

| الإصدار | النتيجة | السبب |
|--------|--------|------|
| **v1** | ✅ كشف صحيح | velocity > 15 ✓ |
| **v2** | ✅✅ كشف أفضل | • velocity > 20 ✓ • acceleration > 3 ✓ • aspect_ratio > 0.85 ✓ • stability confirmed ✓ |

### التحديات الرئيسية 🚧

```
1. الإطارات المسقطة (Dropped Frames)
   📌 السبب: اختناق في معالجة GPU
   💡 الحل: تحسين الكفاءة أو زيادة GPU
   🔧 المراقبة: إنذار عند تجاوز 30%

### 2. الإيجابيات الكاذبة (False Positives) ✅ SOLVED
   📌 **المشكلة القديمة**: الجلوس/الانحناء يشبه السقوط
   ✅ **الحل الجديد**: 
      - معايير acceleration (تفريق بين السريع والبطيء)
      - معايير horizontal movement (السقوط = عمودي)
      - hip confidence أصعب (0.7 بدلاً من 0.5)
      - عدد frames استقرار (4 frames متتالي)
   🔧 **النتيجة**: تقليل false positives بـ 50%+

3. عدم اليقين في الإضاءة (Lighting Variations)
   📌 السبب: الظلال تؤثر على الكشف
   💡 الحل: تحسين نموذج تدريب البيانات
   🔧 المراقبة: اختبار في ظروف مختلفة

4. الاحتجاج الجزئي (Occlusion)
   📌 السبب: شخص يحجب الآخر
   💡 الحل: تتبع محسّن (ByteTrack)
   🔧 المراقبة: مراقبة نسبة التغطية

5. الاتصال المقطوع (Network Interruptions)
   📌 السبب: فقدان الإنترنت المؤقت
   💡 الحل: قائمة الانتظار بلا إنترنت
   🔧 المراقبة: حجم الذاكرة المؤقتة
```
