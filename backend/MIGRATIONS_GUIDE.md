# Database Migrations Guide

This project uses **Alembic** for database schema version control and migrations. Alembic integrates with SQLAlchemy to track and apply database changes safely.

## Quick Start

### Prerequisites

- Alembic is already installed (`alembic==1.13.2` in requirements.txt)
- PostgreSQL database is running
- Environment variables are configured (see `.env.example`)

### Basic Commands

#### Run Pending Migrations (Upgrade)

```bash
cd backend
alembic upgrade head
```

Applies all pending migrations to the database, bringing it to the latest schema version.

#### Revert Last Migration (Downgrade)

```bash
cd backend
alembic downgrade -1
```

Rolls back the most recent migration. You can specify any revision ID or `-N` for N steps back.

#### Check Migration Status

```bash
cd backend
alembic current
```

Displays the current database revision.

```bash
cd backend
alembic history
```

Shows all applied and pending migrations.

#### View the Next Pending Migration

```bash
cd backend
alembic heads
```

Shows all migration branch heads (typically one for a linear history).

## Creating New Migrations

### Automatic Migration (Recommended)

When you modify SQLAlchemy models, generate a migration automatically:

```bash
cd backend
alembic revision --autogenerate -m "Descriptive migration message"
```

For example:
```bash
alembic revision --autogenerate -m "Add status field to Camera model"
```

This creates a new file in `alembic/versions/` with `upgrade()` and `downgrade()` functions that Alembic detected based on model changes.

**Review the generated migration** before applying—autogenerate can sometimes miss or misinterpret changes.

### Manual Migration

For complex schema changes, create an empty migration and write SQL manually:

```bash
cd backend
alembic revision -m "Custom schema update"
```

Then edit the generated file in `alembic/versions/` and add your SQL:

```python
def upgrade() -> None:
    op.add_column('incidents', sa.Column('custom_field', sa.String(255)))

def downgrade() -> None:
    op.drop_column('incidents', 'custom_field')
```

## Configuration

### alembic.ini

Key settings:
- **script_location**: Points to `alembic/` directory
- **sqlalchemy.url**: Uses DATABASE_URL environment variable (see `alembic/env.py`)

### alembic/env.py

- Imports all models from `app.models` for autogenerate detection
- Reads `DATABASE_URL` from environment or `.env` file
- Supports both offline (SQL string generation) and online (direct DB connection) modes

## Workflow Example

1. **Modify a model** in `app/models/models.py`:

   ```python
   class Incident(Base):
       __tablename__ = "incidents"
       id = Column(Integer, primary_key=True)
       title = Column(String(255))
       status = Column(String(50))  # New field
   ```

2. **Generate migration**:

   ```bash
   alembic revision --autogenerate -m "Add status field to Incident"
   ```

3. **Review the generated file** in `alembic/versions/`:

   ```python
   def upgrade() -> None:
       op.add_column('incidents', sa.Column('status', sa.String(50)))

   def downgrade() -> None:
       op.drop_column('incidents', 'status')
   ```

4. **Apply the migration**:

   ```bash
   alembic upgrade head
   ```

5. **Commit to git**:

   ```bash
   git add alembic/versions/<migration_file>.py
   git commit -m "Migration: Add status field to Incident"
   ```

## Docker & Deployment

### Running Migrations in Docker

Add a migration step to your deployment pipeline. In `docker-compose.yml`, you can create an initialization service:

```yaml
backend-migrate:
  image: visionsafe360-backend:latest
  environment:
    - DATABASE_URL=postgresql://...
  command: alembic upgrade head
  depends_on:
    - db
```

Or run manually after container startup:

```bash
docker exec <backend-container> alembic upgrade head
```

### Kubernetes

In your Helm values or deployment, run migrations as a Job before deploying the main application:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
spec:
  template:
    spec:
      containers:
      - name: alembic
        image: visionsafe360-backend:latest
        command: ["alembic", "upgrade", "head"]
        env:
        - name: DATABASE_URL
          valueFrom:
            secretKeyRef:
              name: db-credentials
              key: url
```

## Best Practices

1. **Review autogenerated migrations** before applying—ensure they match your intent
2. **Test migrations locally** with a copy of production data
3. **Write descriptive messages**: `alembic revision --autogenerate -m "Add email uniqueness constraint"`
4. **Keep migrations atomic**: One logical change per migration
5. **Never edit applied migrations** in production; create a new migration to undo/redo
6. **Test both upgrade and downgrade** paths: `alembic downgrade -1 && alembic upgrade head`
7. **Commit migration files to git** alongside model changes
8. **Use version control** to track schema evolution

## Troubleshooting

### Migration Won't Apply

Check the current database state:
```bash
alembic current
alembic heads
```

Ensure the database connection is valid:
```bash
python -c "from app.config.database import DATABASE_URL; print(DATABASE_URL)"
```

### Autogenerate Not Detecting Changes

- Verify models are imported in `alembic/env.py`
- Check that model changes are syntactically correct
- Ensure the database and model schema are in sync (manual sync or force-refresh)

### Downgrade Failures

Some operations (e.g., data-destroying operations) may not have reversible SQL. Review the downgrade function and add custom SQL if needed.
